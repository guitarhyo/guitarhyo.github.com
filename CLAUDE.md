# 프로젝트 가이드

이 파일은 Claude Code가 이 저장소에서 작업할 때 참고하는 가이드입니다.

## 프로젝트 개요

웹 기반 미니 게임 모음 사이트입니다. GitHub Pages로 배포되며, 학습 목적으로 게임마다 다른 프레임워크/라이브러리를 사용해 구현합니다.

- **호스팅**: GitHub Pages
- **목적**: 웹 게임 개발 학습 + 포트폴리오
- **추후 계획**: 완성된 게임은 PWA로 만들어 안드로이드 앱으로도 배포

## 사이트 구조

```
/
├── index.html              # 게임 리스트 (허브 페이지)
├── css/                    # 허브 페이지 스타일
├── js/                     # 허브 페이지 스크립트
├── lang/                   # 허브 페이지 다국어 리소스 (ko.json, en.json)
├── assets/                 # 허브 공용 이미지·아이콘
├── games/
│   ├── [게임이름1]/
│   │   ├── index.html      # 게임 진입점
│   │   ├── css/
│   │   ├── js/
│   │   ├── lang/           # 게임별 다국어 리소스
│   │   ├── assets/
│   │   └── ...
│   └── [게임이름2]/
└── CLAUDE.md
```

- **루트 `index.html`**: 게임 리스트를 보여주는 허브 페이지. 단순한 정적 HTML로 유지(빌드 도구 없음).
- **`games/[게임이름]/`**: 각 게임은 완전히 독립된 폴더. 게임마다 다른 프레임워크를 써도 서로 영향 없음.
- 허브에서 각 게임으로는 상대 경로 링크(`./games/runner/`)로 이동.
- **파일은 항상 종류별 폴더로 분리하여 저장** (CSS, JS, 이미지, 언어 리소스 등).

## 기술 스택 정책

게임마다 다른 빌드 방식/라이브러리를 쓰는 것이 **의도된 학습 목표**입니다. 통일하지 않습니다.

### 구성 요소

게임의 기술 스택은 다음 두 축의 조합으로 결정됩니다.

**빌드 방식 (반드시 택1)**
- A) 순수 HTML/CSS/JS (바닐라) — 빌드 도구 없음
- B) Vue 3 + Vite
- C) React + Vite
- D) 그 외 (Svelte 등)

**게임 라이브러리 (선택, 위 빌드 방식 위에 얹어서 사용)**
- Phaser.js — 2D 게임 엔진 (물리, 스프라이트, 씬 관리 내장)
- PixiJS — 빠른 2D 렌더링 (저수준, 자유도 높음)
- Three.js / react-three-fiber — 3D
- 없음 (DOM/Canvas 직접 제어)

예시 조합: "바닐라 JS + Phaser", "Vue + 라이브러리 없음", "React + PixiJS" 등.

### 새 게임 시작 시 스택 결정 절차

Claude Code는 새 게임 작업을 시작하기 전 다음 순서로 사용자와 협의할 것.

1. **게임 컨셉 파악**: 장르, 핵심 메커니즘, 시각적 스타일, 복잡도를 사용자에게 확인.
2. **추천 스택 제시**: 게임 특성에 맞는 빌드 방식과 라이브러리 조합을 2~3개 추천하고, 각각의 장단점·학습 포인트를 함께 설명.
3. **학습 목적 고려**: 사용자가 이미 만들어본 스택은 가급적 중복 추천하지 않기. 새로운 조합을 우선 제안하여 학습 범위 확장.
4. **사용자 확정 후 진행**: 사용자가 스택을 고르면 그때 코드 작성 시작. 임의로 정해서 진행하지 말 것.

### 게임 유형별 스택 가이드 (참고용)

Claude Code가 추천할 때 참고할 수 있는 일반적인 매칭. 절대 규칙은 아니고 출발점일 뿐.

- **UI 중심 게임** (카드, 턴제, 클리커, 비주얼 노벨, 퍼즐 일부): 프레임워크(Vue/React) 단독으로 충분. 라이브러리 불필요.
- **실시간 액션** (러너, 슈팅, 플랫포머, 물리 기반): Phaser 또는 PixiJS 권장. 매 프레임 리렌더링이 필요해 React/Vue 단독은 부적합.
- **간단한 캐주얼** (탭/스와이프 위주, 가벼운 애니메이션): 바닐라 JS로 충분. 학습 초반에 적합.
- **3D**: Three.js 또는 react-three-fiber.

### 공통 제약

- 어떤 스택을 고르든 아래의 [반응형/입력 정책](#반응형--입력-정책) 준수 (마우스/터치 통합).
- 어떤 스택을 고르든 아래의 [다국어 정책](#다국어i18n-정책) 준수.
- 어떤 스택을 고르든 아래의 [공통 디자인 토큰](#공통-디자인-토큰) 사용 권장.

## 게임별 빌드/배포 규칙

### 순수 HTML/JS 게임
- `games/[이름]/` 폴더에 `index.html`을 바로 배치. 별도 빌드 없음.
- CSS는 `css/`, JS는 `js/` 폴더로 분리.
- Phaser 등 라이브러리는 CDN 또는 로컬 복사본으로 로드.

### Vite 기반 게임 (Vue/React/Svelte)
- 개발은 `games/[이름]-src/` 같은 별도 작업 폴더에서 진행.
- `vite.config.js`에 반드시 다음 설정:
  ```js
  base: '/[저장소이름]/games/[게임이름]/'
  ```
- 빌드 결과(`dist/`)를 `games/[게임이름]/`으로 복사하여 배포.
- 또는 `vite.config.js`의 `build.outDir`을 `'../games/[게임이름]'`으로 지정.

### 게임 라이브러리 사용 시 추가 규칙
- **Phaser/PixiJS/Three.js 등 게임 인스턴스 정리 필수**: SPA 환경(Vue/React)에서 컴포넌트 언마운트 시 `game.destroy()` 호출하여 메모리 누수 방지.
- **에셋 경로는 상대 경로**: GitHub Pages base path를 고려해 `./assets/sprite.png` 형태로 로드.
- **입력 시스템**: 라이브러리 자체의 입력 API가 마우스/터치 통합을 지원하는지 확인. Phaser는 기본 지원.

### 공통
- 모든 경로는 상대 경로 사용 권장 (절대 경로 `/`로 시작 금지).
- 이미지·폰트 등 에셋도 게임 폴더 안에 두기.
- 파일은 종류별로 폴더 분리하여 저장 (한 폴더에 모든 파일을 몰아넣지 않기).

## 다국어(i18n) 정책

모든 게임과 허브 페이지는 **한국어/영어 전환 옵션**이 필수입니다.

### 공통 규칙
- 언어 리소스는 `lang/ko.json`, `lang/en.json` 형태로 분리.
- UI 어딘가에 언어 전환 토글 또는 버튼 배치.
- 사용자 선택 언어는 `localStorage`에 `lang` 키로 저장하여 다음 방문 시 유지.
- 초기 진입 시 저장된 값이 없으면 브라우저 언어(`navigator.language`) 감지하여 적용.

### 프레임워크별 구현
- **바닐라 JS**: `fetch()`로 JSON 로드 → 직접 텍스트 치환. 의존성 없이 가볍게.
- **Vue**: `vue-i18n` 사용.
- **React**: `react-i18next` 또는 가벼운 경우 직접 Context 구현.
- 어느 방식이든 텍스트는 반드시 코드와 분리된 외부 파일(JSON 등)에 둘 것.

## 반응형 / 입력 정책

### 화면
- **모바일 우선(mobile-first) 설계**. 모든 레이아웃은 모바일에서 먼저 잘 보여야 함.
- PC 환경에서는 반응형으로 화면 폭에 맞춰 적응. 게임 영역이 무한정 커지지 않도록 최대 폭 제한 권장.

### 조작 (중요)
- **기본 입력은 마우스/터치 통합**. 같은 코드로 모바일 터치와 PC 마우스 클릭을 모두 처리.
  - 권장: `pointerdown`, `pointermove`, `pointerup` 등 Pointer Events 사용.
  - 클릭 = 탭, 드래그 = 스와이프로 동일하게 동작.
- **키보드는 기본 사용 안 함**. 단, 특정 게임의 게임성에 키보드 조작이 꼭 필요하면 그 게임에 한해 추가 가능 (예: 타자 게임, 슈팅 게임의 발사키 등).
- 키보드를 추가하는 경우라도 **마우스/터치 조작은 반드시 함께 제공**할 것. 키보드 전용 조작은 금지.

## 허브 페이지(index.html) 규칙

- 의존성 없는 순수 HTML/CSS/JS로 유지.
- HTML/CSS/JS 파일을 분리하여 작성 (인라인 스타일·스크립트 최소화).
- 게임 카드 목록 형태. 각 카드는 `./games/[이름]/`으로 링크.
- 새 게임이 추가되면 허브의 카드 목록도 함께 업데이트할 것.
- 모바일 최적화, PC에서는 반응형.
- 언어 전환 토글 포함.

## 점수 저장 정책

- 게임별 최고 점수는 `localStorage`에 게임 단위로 저장.
- 키 네이밍: `score:[게임이름]:[지표명]` (예: `score:runner:best`)
- 게임 간 점수 공유는 하지 않음 (각 게임 독립).
- 서버 저장·랭킹 시스템은 현재 범위 밖.

## 공통 디자인 토큰

허브와 게임들이 어느 정도 시각적 통일감을 갖도록 다음 토큰을 공용으로 사용합니다. 각 게임은 자체 디자인을 더할 수 있지만 베이스는 동일하게.

CSS 변수로 정의 권장 (`:root`에 선언):

```css
:root {
  /* 색상 (예시 — 실제 값은 첫 작업 시 확정) */
  --color-bg: #0f1115;
  --color-surface: #1a1d24;
  --color-text: #e8e8ea;
  --color-accent: #4f8cff;

  /* 폰트 */
  --font-sans: system-ui, -apple-system, "Noto Sans KR", sans-serif;

  /* 간격 */
  --space-sm: 8px;
  --space-md: 16px;
  --space-lg: 24px;

  /* 모서리 */
  --radius-sm: 8px;
  --radius-md: 12px;
}
```

- 게임 안에서 위 변수를 그대로 사용하거나 오버라이드 가능.
- 정확한 색상값·폰트는 첫 작업(허브 페이지) 시 사용자와 확정.

## GitHub Pages 배포

- `main` 브랜치의 루트에서 서빙되는 구조.
- 푸시하면 자동 반영됨.
- HTTPS 기본 지원되므로 PWA 변환도 추후 가능.

## 작업 시 지켜야 할 것

1. **새 게임 추가 시**: 게임 컨셉 확인 → 스택 추천 → 사용자 확정 후 작업 시작 (위 [스택 결정 절차](#새-게임-시작-시-스택-결정-절차) 준수).
2. **base path 항상 확인**: GitHub Pages 서브경로에서 작동해야 하므로 빌드 도구의 base 설정 필수.
3. **허브와 게임 분리 유지**: 허브 페이지에 프레임워크 의존성 추가하지 말 것.
4. **파일 분리 유지**: HTML 한 파일에 CSS·JS를 몰아넣지 말 것. 폴더 구조 규칙 준수.
5. **다국어 누락 금지**: 새 텍스트를 추가할 때마다 `ko.json`, `en.json` 모두 업데이트.
6. **입력 방식 확인**: 마우스/터치로 작동하는지 PC와 모바일 양쪽에서 검증.
7. **게임 라이브러리 정리**: Phaser/PixiJS 등 사용 시 SPA 환경에서 인스턴스 destroy 누락 금지.
8. **커밋 메시지**: 한국어 또는 영어 자유. 단, 변경한 게임을 명시할 것 (예: `[runner] 점프 물리 수정`).
9. **README 업데이트**: 새 게임을 추가하면 루트 `README.md`에도 항목 추가.

## 학습 메모

이 저장소는 학습이 목적이므로, 새 개념이나 패턴을 도입할 때는 코드에 **왜 그렇게 했는지 짧은 주석**을 남겨주세요. 특히 프레임워크별 차이점이 드러나는 부분(상태 관리, 컴포넌트 구조, 렌더링 방식 등)은 학습 포인트가 되므로 설명을 적극적으로 추가할 것.

각 게임 폴더에는 가능하면 짧은 `README.md`를 두어, 이 게임이 어떤 스택으로 만들어졌고 어떤 학습 포인트가 있었는지 기록할 것.

마지막으로 이 코드의 보안 취약점을 OWASP Top 10 기준으로 점검해줘.
